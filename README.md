<h1 align="center">Докога? / Dokoga</h1>

**AI прогноза за просрочването на обществените ремонти в България.**

Питаш: *Докога?* — избираш ремонт на път, улица или парк и AI-ят предсказва
кога РЕАЛНО ще свърши, спрямо обещания срок. Показва и кои изпълнители и
общини просрочват най-много.

> ZaraHack 2026 · Тема: *Find the Hidden Story*

---

## Идеята накратко

Обществените ремонти системно се проточват — обещават 3 месеца, става 9.
`Докога?` открива тази скрита закономерност в данните за обществените поръчки
и я превръща в личен, споделяем инструмент.

- 🔮 **Предсказание** — „Обещано: 3 мес. Реалистично: ~8 мес."
- 🤖 **AI обяснение** — защо се бави (история на изпълнителя, сезон, стойност)
- 🏗️ **Рейтинг на изпълнителите** — кои фирми винаги просрочват
- 🗺️ **Карта** само на гражданско-значимите ремонти (пътища, паркове, ВиК, осветление, площади), оцветени по % просрочване
- 🧠 **AI анализ на проект** — клик на точка → LLM обяснява ЗАЩО ще се бави (граундирано в риска, историята на областта и изпълнителя)
- 💬 **AI чат на български** над проверени данни — пита те няколко въпроса за „кога ще свърши моят ремонт“ и връща груба ML прогноза (с ясен дисклеймър, че е спекулация), а при въпрос за конкретен проект (напр. най-голям бюджет) **плавно завърта картата** натам
- 📲 **Споделяема карта** — резултатът от предиктора става брандиран PNG (Web Share / download)

## Данни

- **Източник:** **ЦАИС ЕОП по стандарт OCDS**, през [data.egov.bg](https://data.egov.bg) → Агенция по обществени поръчки (лиценз CC0)
- **11 двуседмични пакета** = целият публикуван период — **25 980 release-а**, **18 206 уникални договора** (по `ocid`)
- Анексите към договорите (`amendments` → изместен `endDate`) = сигналът за просрочване (ML етикет)
- *(РОП „Договори и изменения" отпадна — само 186 договора, без структуриран срок.)*

## Как е трениран моделът (Data Science)

> Цялата работа е възпроизводима през `scripts/01..06`. Данни: CC0.

### 1. Източник на данните
Първоначалният РОП датасет („Договори и изменения") се оказа неизползваем —
само 186 договора и **без структуриран срок**. Затова източникът е
**ЦАИС ЕОП по стандарт OCDS** (data.egov.bg, Агенция по обществени поръчки):
- **11 двуседмични пакета** — целият 2026 (01-01 → 03-06)
- **25 980 release-а**, **18 206 уникални договора** (по `ocid`)
- сваляне: `scripts/01,04,05` (2-стъпков endpoint, кеширан локално)

### 2. Конструкция на етикета (просрочване)
OCDS дава `contracts.period.startDate/endDate` (планиран срок) и `amendments`.
За всеки `ocid` групираме всички release-и през времето:
- `planned_end` = най-ранно обявеният `endDate`
- `final_end`   = най-късно обявеният `endDate` (след анекси)
- **`overrun_days = final_end − planned_end`**

Резултат: **4 470** договора с планиран срок + стойност; **289 (6.5%)** с
реално удължаване. Works (строителство): 610; ремонти по ключови думи: 369.

### 3. Признаци (features)
`log(стойност)`, `value_per_day` (интензитет на разхода = стойност/срок),
`планиран_срок_дни`, `start_month` (сезон), `region` (NUTS / област),
`category` (works/goods/services), `sector` (гражданско-значим сектор по
заглавие — `scripts/sectors.py`), `is_repair`, `n_tenderers` (конкуренция).
*Без* `has_amendment` — това е изтичане на етикета.

### 4. Модели (LightGBM)
- **Риск модел** — `LGBMClassifier`, 5-fold стратифицирана CV.
  → **ROC-AUC = 0.63**, PR-AUC = 0.14 (baseline 0.065). Умерен, но реален
  сигнал. Топ драйвери: **планиран срок, стойност, интензитет на разхода**.
- **Дни модел** — `LGBMRegressor` върху положителните. Не бие baseline по MAE
  (289 примера са малко), но **варира смислено по стойност/срок/регион/сезон**.
  „Очаквани дни" се сервират като blend (0.7·регресор + 0.3·медиана по сектор),
  клампнат в [7, 900] — затова всяка точка/прогноза е различна, не константа.
- **Защо само 2026:** структуриран срок (планиран край) има САМО в OCDS, който
  АОП публикува от 01-2026. Историческият РОП (2024/2025) носи стойност и анекси,
  но **няма поле за срок** → не може да се етикетира просрочване. Затова моделът
  дава честна **долна граница**, не присъда.

### 5. Честни ограничения
- Публикуван е **само 2026** (5 месеца) → скорошните договори още не са
  анексирани, значи просрочванията са **подценени** (моделът дава долна граница).
- 289 положителни примера → рисковият скор е **proof-of-concept**, не
  продукционен. Показваме несигурност, не твърдим точност, която нямаме.
- Подобрения с повече история: история на изпълнителя по ЕИК, CPV кодове,
  кръстосване с ИСУН за реални дати на завършване.

### 6. Възпроизвеждане
```bash
python scripts/01_download.py      # РОП (за сравнение)
python scripts/04_discover_ocds.py # изброява OCDS датасетите
python scripts/05_build_dataset.py # сглобява етикета -> data/processed/works.csv
python scripts/06_train.py         # тренира -> models/ + metrics.json
```

## Backend (LLM слой)

AI чат на български над данните — с **tool-use граундинг**: LLM-ът никога не
смята числа сам, само разказва проверени стойности от `projects.sqlite`.

### Архитектура: retrieve → narrate
1. **route** — моделът избира една заявка от въпроса (JSON `{tool, arg}`).
2. **retrieve** — `tools.py` изпълнява детерминистична SQL заявка → верни числа.
3. **narrate** — моделът разказва САМО тези числа на български.

Избран е този модел (а не native function calling), защото Gemma често не
поддържа tools — така граундингът е гарантиран при всеки модел.

### Файлове (`backend/`)
- `tools.py` — 6 SQLite инструмента: `get_region_stats`, `get_contractor_stats`,
  `search_contracts`, `top_by_value` (най-голям бюджет), `top_risky_regions`,
  `top_risky_contractors`. Отговорът връща и `focus` (ocid/област) → картата отлита натам.
- `agent.py` — `route → retrieve → narrate` (чат) + `analyze()` (граундиран
  анализ на конкретен проект: риск + дни + рискови фактори + статистика на
  областта/изпълнителя → LLM разказва защо ще се бави).
- `predictor.py` — LightGBM за `/predict` (риск + вариращи очаквани дни + рискови фактори).
- `serve.py` — FastAPI: `POST /chat`, `POST /predict`, `POST /analyze`, `GET /health`.
- `.env` — `GOOGLE_API_KEY` (в .gitignore, не се качва).

### Модели (Google GenAI)
Главен: **`gemini-3.1-flash-lite-preview`** (по-бърз) · fallback: `gemma-4-31b-it`,
`gemma-4-26b-a4b-it`. При грешка се опитва следващия по реда.

### Защити
- **Rate limit** (30 заявки/мин на IP) на `/chat` и `/predict`.
- **CORS allowlist** (конфигурируем чрез `ALLOWED_ORIGINS`).
- Input cap (≤500 знака), премахване на контролни символи, security headers
  (`X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`).
- LLM-ът не смята числа → невъзможно да халюцинира суми (граундинг).

## Frontend (React + Vite)

„Контролна зала на пътен ремонт" — цяла тъмна карта на България с рискови
точки, върху нея плаващи glass HUD панели и AI чат.

### Стек
- **React + Vite + TypeScript** · **react-router** (`/` landing, `/app` карта)
- **Leaflet** (react-leaflet) + CARTO плочки (light/dark) → 1029 ремонта от `projects.geojson`, оцветени по **континуален** риск-градиент (зелено→жълто→оранж→червено)
- **Motion** за анимации (с `prefers-reduced-motion`)
- **Phosphor** иконки · self-host шрифтове (Unbounded / Manrope / JetBrains Mono, пълна кирилица)

### Дизайн
Асфалтов фон + safety-orange сигнатура. Сигнатурен елемент: **delay bar**
(обещан срок + щриховано просрочване, завършващо с „?"). Frosted-glass HUD панели
само върху живата карта (целево, с reduced-transparency fallback).
**Light + dark тема** (токени, превключвател, по подразбиране тъмна), контраст
≥4.5:1, една радиус- и цветова скала. Custom select (не native white-on-white).
Дизайнът минава през skill-овете frontend-design, ui-ux-pro-max, impeccable, taste.

### Екрани
- **Landing** (`/`) — hero, статистики, „Как работи", интерактивен предиктор, footer.
- **Карта** (`/app`) — ляв rail (бранд, филтър по 28 области, **филтър по вид ремонт** — пътища/паркове/ВиК/осветление/площади, легенда); клик на точка → детайл-карта с прогноза „обещано → реалистично", рискови фактори и бутон **AI анализ**; AI чат док → `/chat`. Чатът прави **разговорна прогноза** (пита вид/област/мащаб/срок → ML отговор + дисклеймър) и **завърта картата** (`flyTo`) към проекта от отговора.
- **Предиктор** — въвеждаш ремонт (вид, област, стойност, срок, оферти, месец) → `/predict` връща риск % + вариращи очаквани дни + рискови фактори, с бутон **Сподели** (брандиран PNG за вирусно споделяне).
- **Метод и почтеност** (на landing-а) — видими секции за източника, конструкцията на етикета, долната граница и AI граундинга, за да е прозрачно всяко число пред журито.

### Стартиране
```bash
cd frontend
npm install
npm run dev        # http://localhost:5173 (backend на :8000 за чата)
```
Картата работи и без backend (статичен `projects.geojson`); чатът иска backend-а.

на риска, не присъда за конкретна фирма.
