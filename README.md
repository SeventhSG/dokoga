# Докога? / Dokoga

**AI прогноза за просрочването на обществените ремонти в България.**

Питаш: *Докога?* — избираш ремонт на път, улица или парк и AI-ят предсказва
кога РЕАЛНО ще свърши, спрямо обещания срок. Показва и кои изпълнители и
общини просрочват най-много.

> ZaraHack 2026 · Тема: *Find the Hidden Story*

---

## Идеята накратко

Обществените ремонти системно се проточват — обещават 3 месеца, става 9.
`Докога?` открива тази скрита закономерност в данните за обществените поръчки
и я превръща в личен, споделяем инструмент.

- 🔮 **Предсказание** — „Обещано: 3 мес. Реалистично: ~8 мес."
- 🤖 **AI обяснение** — защо се бави (история на изпълнителя, сезон, стойност)
- 🏗️ **Рейтинг на изпълнителите** — кои фирми винаги просрочват
- 🗺️ **Карта** на ремонтите, оцветени по % просрочване
- 💬 **AI чат на български** над проверени данни

## Данни

- **Източник:** [data.egov.bg](https://data.egov.bg) → Агенция по обществени поръчки (лиценз CC0)
- **Договори и изменения на договори** — [2024](https://data.egov.bg/data/view/88ea1672-944b-4b9a-b074-528e316eab46) · [2025](https://data.egov.bg/data/view/7990cb41-719d-4616-b656-c750ebb487d7)
- Изменениятa на договорите (анекси) = сигналът за просрочване (ML етикет)
- ЦАИС ЕОП (OCDS стандарт) за структуриран срок на договора

## Stack

Python (pandas, lightgbm) · Next.js · Leaflet/MapLibre · Claude API (tool-use)

## Pipeline

```
CSV (вход) → pandas (модел + сливане) → GeoJSON (карта) + SQLite (AI/заявки)
```

## Как е трениран моделът (Data Science)

> Цялата работа е възпроизводима през `scripts/01..06`. Данни: CC0.

### 1. Източник на данните
Първоначалният РОП датасет („Договори и изменения") се оказа неизползваем —
само 186 договора и **без структуриран срок**. Затова източникът е
**ЦАИС ЕОП по стандарт OCDS** (data.egov.bg, Агенция по обществени поръчки):
- **11 двуседмични пакета** — целият 2026 (01-01 → 03-06)
- **25 980 release-а**, **18 206 уникални договора** (по `ocid`)
- сваляне: `scripts/01,04,05` (2-стъпков endpoint, кеширан локално)

### 2. Конструкция на етикета (просрочване)
OCDS дава `contracts.period.startDate/endDate` (планиран срок) и `amendments`.
За всеки `ocid` групираме всички release-и през времето:
- `planned_end` = най-ранно обявеният `endDate`
- `final_end`   = най-късно обявеният `endDate` (след анекси)
- **`overrun_days = final_end − planned_end`**

Резултат: **4 470** договора с планиран срок + стойност; **289 (6.5%)** с
реално удължаване. Works (строителство): 610; ремонти по ключови думи: 369.

### 3. Признаци (features)
`log(стойност)`, `планиран_срок_дни`, `start_month` (сезон), `region` (NUTS /
област), `category` (works/goods/services), `is_repair` (ключови думи),
`n_tenderers` (конкуренция). *Без* `has_amendment` — това е изтичане на етикета.

### 4. Модели (LightGBM)
- **Риск модел** — `LGBMClassifier`, 5-fold стратифицирана CV.
  → **ROC-AUC = 0.637**, PR-AUC = 0.138 (baseline 0.065). Умерен, но реален
  сигнал. Топ драйвери: **стойност, планиран срок, сезон**.
- **Дни модел** — `LGBMRegressor` върху положителните. MAE 123 vs baseline 117
  → **не бие baseline** (289 примера са малко). Затова „очаквани дни" се
  поднасят като **историческа медиана по сегмент** (works = 63 дни), не като
  предсказание.

### 5. Честни ограничения
- Публикуван е **само 2026** (5 месеца) → скорошните договори още не са
  анексирани, значи просрочванията са **подценени** (моделът дава долна граница).
- 289 положителни примера → рисковият скор е **proof-of-concept**, не
  продукционен. Показваме несигурност, не твърдим точност, която нямаме.
- Подобрения с повече история: история на изпълнителя по ЕИК, CPV кодове,
  кръстосване с ИСУН за реални дати на завършване.

### 6. Възпроизвеждане
```bash
python scripts/01_download.py      # РОП (за сравнение)
python scripts/04_discover_ocds.py # изброява OCDS датасетите
python scripts/05_build_dataset.py # сглобява етикета -> data/processed/works.csv
python scripts/06_train.py         # тренира -> models/ + metrics.json
```

## Backend (LLM слой)

AI чат на български над данните — с **tool-use граундинг**: LLM-ът никога не
смята числа сам, само разказва проверени стойности от `projects.sqlite`.

### Архитектура: retrieve → narrate
1. **route** — моделът избира една заявка от въпроса (JSON `{tool, arg}`).
2. **retrieve** — `tools.py` изпълнява детерминистична SQL заявка → верни числа.
3. **narrate** — моделът разказва САМО тези числа на български.

Избран е този модел (а не native function calling), защото Gemma често не
поддържа tools — така граундингът е гарантиран при всеки модел.

### Файлове (`backend/`)
- `tools.py` — 5 SQLite инструмента: `get_region_stats`, `get_contractor_stats`,
  `search_contracts`, `top_risky_regions`, `top_risky_contractors`.
- `agent.py` — `route → retrieve → narrate` + fallback верига от модели.
- `serve.py` — FastAPI: `POST /chat {message}` и `GET /health`.
- `.env` — `GOOGLE_API_KEY` (в .gitignore, не се качва).

### Модели (Google GenAI)
Главен: `gemma-4-31b-it` · fallback: `gemini-3.1-flash-lite-preview`,
`gemma-4-26b-a4b-it`. При грешка се опитва следващия по реда.

### Стартиране
```bash
cd backend
pip install -r requirements.txt
echo "GOOGLE_API_KEY=..." > .env
uvicorn serve:app --reload          # HTTP сервиз
python agent.py "въпрос"            # CLI тест
python test_endpoint.py            # тест на /chat
```

## Frontend (React + Vite)

„Контролна зала на пътен ремонт" — цяла тъмна карта на България с рискови
точки, върху нея плаващи glass HUD панели и AI чат.

### Стек
- **React + Vite + TypeScript**
- **Leaflet** (react-leaflet) + тъмни CARTO плочки → 1029 ремонта от `projects.geojson`, оцветени по риск (зелено/жълто/червено)
- **Motion** за входни анимации (с `prefers-reduced-motion`)
- **Phosphor** иконки · self-host шрифтове (Unbounded / Manrope / JetBrains Mono, пълна кирилица)

### Дизайн
Асфалтов тъмен фон + safety-orange сигнатура. Сигнатурен елемент: **delay bar**
(обещан срок + щриховано просрочване, завършващо с „?"). Frosted-glass HUD панели
само върху живата карта (целево, с reduced-transparency fallback). Заключена тъмна
тема, контраст ≥4.5:1, една радиус- и цветова скала.

### Екрани
- Ляв rail: бранд, филтър по област (28 области), легенда за риска
- Клик на точка → детайл-карта (риск %, очаквано забавяне, стойност, изпълнител)
- AI чат док → вика backend `/chat`, разказва само проверени числа

### Стартиране
```bash
cd frontend
npm install
npm run dev        # http://localhost:5173 (backend на :8000 за чата)
```
Картата работи и без backend (статичен `projects.geojson`); чатът иска backend-а.

## Етика

Изходът е „очаквано забавяне по исторически данни", не обвинение в измама.
Източниците се цитират, несигурността се показва. Моделът дава **долна граница**
на риска, не присъда за конкретна фирма.
