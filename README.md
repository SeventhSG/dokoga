## Скритата история

Обществените ремонти системно се проточват: обещават 3 месеца, става 9.
`Докога?` открива тази закономерност в данните за обществените поръчки и я
превръща в личен, споделяем инструмент, който всеки гражданин може да провери
за своята улица.

Куката е лична: всеки има омразен ремонт пред блока си.

## Възможности

| Функция | Какво прави |
|---|---|
| 🔮 **Предиктор** | Въвеждаш вид/област/стойност/срок и получаваш риск % + очаквано забавяне в дни (вариращо, не константа) + рискови фактори |
| 🗺️ **Карта** | Само гражданско-значимите ремонти (пътища, паркове, ВиК, осветление, площади), оцветени по континуален риск-градиент; размерът на точката е по стойността |
| 🧠 **AI анализ на проект** | Клик на точка → LLM обяснява ЗАЩО ще се бави, граундирано в риска, рисковите фактори и историята на областта/изпълнителя |
| 💬 **AI чат на български** | Питаш за област, изпълнител или най-голям бюджет; отговаря само с проверени числа |
| 🗣️ **Разговорна прогноза** | „Кога ще свърши ремонтът пред блока ми?" → чатът пита няколко въпроса и връща груба ML прогноза с ясен дисклеймър, че е спекулация |
| 🧭 **Навигация от чата** | Питаш за конкретен проект → картата плавно завърта (`flyTo`) към него и отваря детайла |
| 📲 **Споделяема карта** | Резултатът от предиктора става брандиран PNG (Web Share / download) за вирусно споделяне |
| 🌗 **Light / dark тема** | Контролна зала по подразбиране тъмна, с превключвател |

## Данни

- **Източник:** ЦАИС ЕОП по стандарт **OCDS**, през [data.egov.bg](https://data.egov.bg) (Агенция по обществени поръчки), лиценз **CC0**.
- **11 двуседмични пакета** = целият публикуван период (01-01-2026 → 03-06-2026): **25 980 release-а**, **18 206 уникални договора** (по `ocid`).
- Анексите (`amendments`, изместен `endDate`) са сигналът за просрочване, който ползваме като ML етикет.
- **Защо не РОП:** РОП „Договори и изменения" (2024/2025) има стойност и анекси, но **няма структуриран срок (планиран край)**, затова от него не може да се сметне просрочване. Структуриран срок има само в OCDS, който АОП публикува от 01-2026. Това е честно документирано: моделът дава **долна граница**, не присъда.

## Машинно обучение (Data Science)

> Възпроизводимо през `scripts/01..07`. Данни: CC0.

### 1. Етикет (просрочване)
OCDS дава `contracts.period.startDate/endDate` (планиран срок) и `amendments`.
За всеки `ocid` групираме release-ите през времето:
- `planned_end` = най-ранно обявеният `endDate`
- `final_end` = най-късно обявеният `endDate` (след анекси)
- **`overrun_days = final_end - planned_end`**

Резултат: **4 470** договора с планиран срок и стойност, от които **289 (6.5%)** с реално удължаване.

### 2. Сектори (гражданско-значими)
`scripts/sectors.py` класифицира всеки договор по заглавие в гражданско-значим
сектор и изхвърля нерелевантните за хората (покриви, енергийна ефективност,
асансьори, офиси, болнични интериори, чисти доставки/услуги). На картата
остават само секторите, които засягат публичното пространство:

| Сектор | Точки на картата |
|---|---|
| Пътища и тротоари | 398 |
| Паркове и площадки | 143 |
| ВиК | 112 |
| Обществени сгради и площади | 42 |
| Улично осветление | 15 |
| **Общо на картата** | **710** |

Моделът обаче се тренира върху **всички 4 470** договора (повече сигнал), а
секторът влиза като признак.

### 3. Признаци (features)
`log(стойност)`, `value_per_day` (интензитет на разхода = стойност/срок),
`планиран_срок_дни`, `start_month` (сезон), `region` (NUTS / област),
`category` (works/goods/services), `sector`, `is_repair`, `n_tenderers`
(конкуренция). Без `has_amendment`, защото това е изтичане на етикета.

### 4. Модели (LightGBM)

| Модел | Резултат |
|---|---|
| Риск (`LGBMClassifier`, 5-fold CV) | ROC-AUC **0.63**, PR-AUC **0.14** (baseline 0.065). Топ драйвери: планиран срок, стойност, интензитет на разхода |
| Дни (`LGBMRegressor` върху позитивите) | Не бие baseline по MAE (289 примера са малко), но **варира** по стойност/срок/регион/сезон |

„Очаквани дни" се сервират като blend `0.7·регресор + 0.3·медиана по сектор`,
клампнат в `[7, 900]`, затова всяка прогноза е различна, а не константа за
сегмента. Рисковите фактори (защо) се генерират от правила върху същите признаци.

### 5. Честни ограничения
- Публикуван е само 2026: скорошните договори още не са анексирани, значи просрочванията са **подценени** (долна граница).
- 289 позитива означава, че рисковият скор е **proof-of-concept**, не продукционен. Показваме несигурност вместо да твърдим точност, която нямаме.
- Възможни подобрения с повече история: профил на изпълнителя по ЕИК, CPV кодове, кръстосване с ИСУН за реални дати на завършване.

## Backend (LLM слой)

AI чат на български над данните, с **retrieve → narrate граундинг**: LLM-ът
никога не смята числа сам, само разказва проверени стойности от `projects.sqlite`.

### Архитектура
1. **route** - моделът избира една заявка от въпроса (JSON `{tool, arg}`).
2. **retrieve** - `tools.py` изпълнява детерминистична SQL заявка и връща верни числа.
3. **narrate** - моделът разказва САМО тези числа на български.

Избран е този подход (а не native function calling), защото Gemma често не
поддържа tools, така граундингът е гарантиран при всеки модел.

### Endpoints
- `POST /chat` - чат (route → retrieve → narrate); връща и `focus` (ocid/област) за навигация на картата.
- `POST /predict` - LightGBM прогноза (риск + вариращи дни + рискови фактори).
- `POST /analyze` - граундиран LLM анализ на конкретен проект (риск + фактори + статистика на областта/изпълнителя → защо ще се бави).
- `GET /health` - проверка.

### Инструменти (`tools.py`)
`get_region_stats`, `get_contractor_stats`, `search_contracts`,
`top_by_value` (най-голям бюджет), `top_risky_regions`, `top_risky_contractors`.

### Модели (Google GenAI)
Главен: `gemini-3.1-flash-lite-preview` (по-бърз) · fallback: `gemma-4-31b-it`,
`gemma-4-26b-a4b-it`. При грешка опитва следващия по реда.

### Защити
- Rate limit (30 заявки/мин на IP) на `/chat`, `/predict`, `/analyze`.
- CORS allowlist (конфигурируем чрез `ALLOWED_ORIGINS`).
- Input cap (≤500 знака), премахване на контролни символи, security headers (`X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`).
- LLM-ът не смята числа, значи не може да халюцинира суми.

## Frontend (React + Vite)

„Контролна зала на пътен ремонт": цяла тъмна карта на България с рискови точки,
върху нея плаващи glass HUD панели и AI чат.

### Екрани
- **Landing** (`/`): hero с живата delay bar, статистики с count-up анимация, „Как работи", секция **AI, на който можеш да вярваш** (защо граундинг + защо LightGBM + честно за границите), секция **Метод и почтеност** (източник, етикет, долна граница, етика), интерактивен предиктор и секция **Отвъд уикенда** (как живее проектът след хакатона).
- **Карта** (`/app`): ляв rail с бранд, филтър по 28 области, **филтър по вид ремонт** (пътища/паркове/ВиК/осветление/площади) и легенда. Клик на точка отваря детайл-карта с прогноза „обещано → реалистично", рискови фактори и бутон **AI анализ**. AI чат док прави **разговорна прогноза** и **завърта картата** към проекта от отговора.
- **Предиктор**: вид, област, стойност, срок, оферти, месец → `/predict` връща риск % + вариращи дни + рискови фактори, с бутон **Сподели** (брандиран PNG).

### Дизайн
Асфалтов фон + safety-orange сигнатура. Сигнатурен елемент: **delay bar**
(обещан срок + щриховано просрочване, завършващо с „?"). Frosted-glass HUD само
върху живата карта (с reduced-transparency fallback). Light + dark тема (токени,
превключвател, по подразбиране тъмна), контраст ≥4.5:1. Self-host шрифтове
(Unbounded дисплей, Manrope текст, JetBrains Mono числа, пълна кирилица).
Иконки: Phosphor. Анимации: Motion (с `prefers-reduced-motion`).

### Стартиране
```bash
cd frontend
npm install
npm run dev        # http://localhost:5173 (backend на :8000 за чата/прогнозата)
```
Картата работи и без backend (статичен `projects.geojson`); чатът, предикторът и AI анализът искат backend-а.

## Етика

Изходът е „очаквано забавяне по исторически данни", не обвинение в измама.
Източниците се цитират, несигурността се показва. Моделът дава **долна граница**
на риска, не присъда за конкретна фирма. Разговорната прогноза винаги се
поднася с ясен дисклеймър, че е спекулация, не точни данни за конкретния обект.
